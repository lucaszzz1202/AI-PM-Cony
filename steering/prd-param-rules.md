# 参数规则参考库

## 定位

本文件是功能完备性检查的辅助参考，不替代 `prd-phases.md` 中的 7 维度原则驱动检查。

**使用方式**：Agent 先按 7 维度原则自主推理每个功能需要的参数清单，然后用本文件做交叉验证——"我推理出来的参数，有没有漏掉这个类型通常需要的？"

**加载时机**：阶段二/三生成问卷前、参数补充检测时、质量自检 §1.5 时。

**重要说明**：
- 本文件是参考而非穷举，实际场景可能需要更多或更少的参数
- Agent 仍需结合具体业务场景判断哪些参数适用
- 如果某个参数在具体场景中确实不适用，无需强制要求

## 一、按触发类型的常见必要参数

当识别出某个功能的触发方式后，参照对应类型检查是否遗漏常见参数。

### 定时触发

| 参数 | 说明 | 示例 |
|------|------|------|
| 执行周期 | 多久执行一次 | 每 5 分钟 / 每天凌晨 2:00 / cron 表达式 |
| 时区 | 定时基于哪个时区 | UTC+8 / 服务器本地时间 |
| 执行窗口 | 是否限制在特定时间段内执行 | 仅工作日 9:00-18:00 |
| 并发控制 | 上一次未完成时新一次是否启动 | 跳过 / 排队 / 强制终止上一次 |
| 失败重试 | 执行失败后的重试策略 | 重试 3 次，间隔 30s 指数退避 |
| 超时处理 | 单次执行超时阈值及处理 | 超过 5 分钟强制终止并告警 |
| 数据范围 | 每次执行处理的数据范围 | 全量 / 增量（基于上次执行时间戳） |
| 首次执行 | 首次部署时是否立即执行 | 立即执行 / 等待下一个周期 |

### 用户操作触发

| 参数 | 说明 | 示例 |
|------|------|------|
| 操作入口 | 用户从哪里触发 | 按钮 / 菜单项 / 快捷键 / 手势 |
| 前置校验 | 触发前需要满足什么条件 | 表单必填项校验 / 权限检查 / 状态检查 |
| 防重复提交 | 如何防止用户重复触发 | 按钮 loading / 幂等 key / 时间窗口去重 |
| 操作确认 | 是否需要二次确认 | 危险操作弹窗确认 / 无需确认 |
| 即时反馈 | 触发后的 UI 反馈 | Toast 提示 / 页面跳转 / 局部刷新 |
| 异步处理 | 操作是同步还是异步 | 同步等待 / 异步轮询 / WebSocket 推送 |
| 撤销能力 | 操作后是否可撤销 | 支持 N 秒内撤销 / 不可撤销 |

### 外部回调触发

| 参数 | 说明 | 示例 |
|------|------|------|
| 回调来源 | 谁发起回调 | 支付网关 / 第三方系统 / Webhook |
| 协议与格式 | 回调的通信协议和数据格式 | HTTPS POST + JSON / MQ 消息 |
| 签名验证 | 如何验证回调合法性 | HMAC-SHA256 签名 / IP 白名单 |
| 幂等处理 | 重复回调如何处理 | 基于唯一 ID 去重 |
| 超时与重试 | 回调方的重试策略（需要我方兼容） | 对方 3 次重试，间隔递增 |
| 响应要求 | 需要返回什么给回调方 | HTTP 200 + success / 业务状态码 |
| 失败通知 | 回调处理失败时的通知机制 | 告警到企微群 / 写入失败队列人工处理 |

### 数据变更触发

| 参数 | 说明 | 示例 |
|------|------|------|
| 监听对象 | 监听哪个数据实体的变更 | 订单表 / 库存字段 / 配置项 |
| 变更类型 | 哪些变更类型触发 | 新增 / 修改 / 删除 / 特定字段变更 |
| 触发条件 | 变更满足什么条件才触发 | 库存低于安全值 / 状态从 A 变为 B |
| 批量处理 | 批量变更时的处理策略 | 逐条触发 / 合并为一次批量处理 |
| 延迟执行 | 是否需要延迟或防抖 | 变更后等待 5s 防抖再执行 |
| 事务边界 | 触发动作与数据变更是否在同一事务 | 同事务 / 异步解耦 |

### 系统自动触发

| 参数 | 说明 | 示例 |
|------|------|------|
| 触发条件 | 系统判断的具体条件 | 连续 3 次健康检查失败 / 队列积压超过阈值 |
| 检测频率 | 系统多久检测一次条件 | 实时 / 每分钟轮询 |
| 阈值定义 | 触发阈值的具体数值 | CPU > 80% 持续 5 分钟 |
| 冷却期 | 触发后多久内不再重复触发 | 10 分钟内不重复告警 |
| 升级机制 | 持续触发时是否升级处理 | 第 1 次告警 → 第 3 次通知主管 → 第 5 次自动降级 |

## 二、按功能类型的常见必要参数

当识别出某个功能属于哪种类型后，参照对应类型检查是否遗漏常见参数。

### CRUD（增删改查）

| 参数 | 说明 |
|------|------|
| 创建：必填字段与默认值 | 哪些字段必填，哪些有默认值 |
| 创建：唯一性约束 | 哪些字段不允许重复 |
| 查询：筛选条件 | 支持哪些筛选维度 |
| 查询：排序规则 | 默认排序字段和方向 |
| 查询：分页策略 | 每页条数、最大页数 |
| 查询：模糊搜索范围 | 搜索覆盖哪些字段 |
| 修改：可编辑字段 | 哪些字段允许修改，哪些创建后不可改 |
| 修改：并发控制 | 多人同时编辑如何处理（乐观锁 / 悲观锁） |
| 删除：删除策略 | 物理删除 / 软删除 |
| 删除：关联数据处理 | 有关联数据时能否删除，关联数据如何处理 |
| 批量操作 | 是否支持批量创建/修改/删除，上限数量 |

### 审批流

| 参数 | 说明 |
|------|------|
| 审批层级 | 几级审批，各级审批人角色 |
| 审批规则 | 什么条件走什么审批路径（如金额分级） |
| 会签/或签 | 多人审批时是全部通过还是任一通过 |
| 驳回处理 | 驳回后回到哪一步（发起人 / 上一级 / 指定节点） |
| 撤回规则 | 发起人在什么阶段可以撤回 |
| 超时处理 | 审批人超时未处理的策略（催办 / 自动转交 / 自动通过） |
| 代理审批 | 审批人不在时是否支持委托 |
| 审批意见 | 是否必填，字数限制 |

### 数据同步

| 参数 | 说明 |
|------|------|
| 同步方向 | 单向 / 双向 |
| 同步频率 | 实时 / 准实时 / 定时批量 |
| 数据映射 | 源字段与目标字段的映射关系 |
| 冲突策略 | 双向同步时数据冲突如何解决（以源为准 / 以时间戳为准 / 人工介入） |
| 增量标识 | 如何识别增量数据（时间戳 / 版本号 / 变更日志） |
| 失败补偿 | 同步失败后的补偿机制（重试 / 人工重跑 / 告警） |
| 数据校验 | 同步后是否做一致性校验 |
| 历史数据 | 首次同步时历史数据如何处理 |

### 报表 / 看板

| 参数 | 说明 |
|------|------|
| 数据源 | 从哪些表/接口取数 |
| 统计维度 | 按什么维度聚合（时间、组织、类型...） |
| 时间范围 | 默认展示范围，最大可查范围 |
| 刷新策略 | 实时计算 / 定时预计算 / 手动刷新 |
| 导出能力 | 是否支持导出，格式（Excel / PDF / CSV） |
| 数据权限 | 不同角色看到的数据范围 |
| 空数据展示 | 无数据时的展示方式 |

### 通知 / 消息

| 参数 | 说明 |
|------|------|
| 通知渠道 | 站内信 / 邮件 / 短信 / 企微 / Push |
| 通知对象 | 固定角色 / 动态计算（如审批人） |
| 通知模板 | 消息内容模板和变量 |
| 发送时机 | 事件触发后立即 / 合并发送（如每小时汇总） |
| 免打扰 | 是否有免打扰时段设置 |
| 已读追踪 | 是否需要追踪已读状态 |
| 发送失败 | 失败后的重试和降级策略 |

### 导入 / 导出

| 参数 | 说明 |
|------|------|
| 文件格式 | Excel / CSV / JSON / XML |
| 模板 | 是否提供导入模板下载 |
| 数据校验 | 导入时的校验规则（格式、必填、唯一性、关联存在性） |
| 错误处理 | 部分失败时的策略（全部回滚 / 跳过错误行 / 生成错误报告） |
| 数据量限制 | 单次导入/导出的最大行数 |
| 异步处理 | 大数据量时是否异步处理 + 进度展示 |
| 重复处理 | 导入数据与已有数据重复时的策略（覆盖 / 跳过 / 报错） |

## 三、使用指引

### 在问卷生成阶段（阶段二/三）

1. Agent 按 7 维度原则推理出功能的参数需求
2. 识别该功能的触发类型和功能类型
3. 对照本文件对应类型的参数表，检查是否有常见参数被遗漏
4. 遗漏的参数纳入问卷问题（仍以"推荐值 + 确认"形式）

### 在参数补充检测阶段

1. 读取用户答案后，识别答案中隐含的触发类型和功能类型
2. 对照本文件检查用户答案是否产生了新的未定义参数
3. 遗漏的参数纳入参数补充问卷
4. **注意：参数补充问卷每阶段最多 1 轮**（与矛盾澄清的 2 轮上限独立计数，详见 `prd-phases.md` 参数补充问卷机制）。如果 1 轮后仍有未定义参数，写入 understanding.md 的"开放问题"标注 `🔴 关键参数未定义`

### 在质量自检 §1.5 阶段

1. 对每个 FR，识别其触发类型和功能类型
2. 对照本文件对应类型的参数表，检查 PRD 中是否已定义
3. 常见必要参数缺失且未标注 `[待确认]` 的，标记为 ⚠️ 建议补充
4. 仍以原有的原则判断（"开发能否直接编码"）为最终裁定标准
