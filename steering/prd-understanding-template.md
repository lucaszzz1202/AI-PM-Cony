# 理解积累文档规范

## 用途

- 跨阶段保持对需求的连贯理解
- 用户确认 Agent 理解是否准确的锚点
- 最终撰写 PRD 的核心输入
- 记录理解演进过程，可追溯

## 更新原则

### 先推导，再验证，最后才问新问题

每个阶段开始前，Agent 应：
1. 读取当前 understanding.md
2. 基于已有信息 + 用户原始输入，尽可能多地推导新内容
3. 把推导结果写入"假设与推导"区域
4. 问卷中只问：推导不出来的 + 推导了但不确定的
5. 用户回答后，将确认的推导移入"已明确"，修正错误的推导

### 增量更新，禁止覆盖

**这是最重要的原则：每次更新 understanding.md 时，必须是增量追加，严禁覆盖或删除已有内容。**

具体规则：
1. **已明确区域**：只能追加新条目，不能删除或修改已有条目（除非用户明确要求修改某个决策）
2. **假设与推导区域**：已确认的假设移入"已明确"后，原位置标注 `✅ 已确认，移入已明确`，不要直接删除
3. **每次更新前**：先完整读取当前 understanding.md 的全部内容，确认所有已有内容都保留
4. **每次更新后**：对比更新前后的"已明确"条目数量，确保只增不减
5. **如果需要修改已有内容**：必须在变更记录中说明修改了什么、为什么修改

违反此原则会导致前序阶段的理解丢失，严重影响最终 PRD 质量。

### 工具使用约束

**更新 understanding.md 时，禁止使用 fsWrite（整文件覆盖写入）。** 必须使用以下方式之一：
- `strReplace`：替换特定区域（如更新版本号、修改假设状态）
- `fsAppend`：在文件末尾追加新内容（如追加变更记录）
- `editCode` 的 `replace_in_node` 或 `insert_node`：精确定位修改

这从工具层面防止意外覆盖已有内容。唯一例外是输入获取阶段首次创建 understanding.md 时，可以使用 fsWrite。

### 版本管理

- 每次更新在文档顶部标注当前版本
- 底部变更记录追加本次变更摘要
- 版本号格式：`v{阶段}.{修订}`，如 v1.0（阶段一首次）、v1.1（阶段一澄清后）、v2.0（阶段二完成）

## 模板

```markdown
# 需求理解文档

> 版本: v{N}.{R}
> 最后更新: {日期} — 阶段{N}完成后
> 复杂度: {低/中/高}

## 产品概要

{一段话描述：这个产品是什么、为谁服务、解决什么核心问题、怎么衡量成功}

## 已明确

{按主题分组，每条标注来源}

### 产品定位
- ... `[来源: 输入直接纳入]` 或 `[来源: 阶段一问卷]`

### 目标用户
- ...

### 核心场景
- ...

### 技术方案
- ...

### 非功能要求
- ...

> **来源标注规范**：
> - `[来源: 输入直接纳入]` — 覆盖度分析为 ✅，从用户输入中直接提取，未经问卷验证
> - `[来源: 阶段N问卷]` — 通过问卷确认的决策
> - `[来源: 阶段N澄清]` — 通过矛盾澄清确认的决策
> - `[来源: Agent推导]` — Agent 基于已有信息推导，用户确认后纳入
> - `[来源: 用户主动补充]` — 用户在问卷之外主动提供的信息

## 假设与推导

{Agent 基于已有信息做出的合理推导，尚未经用户确认}

- ⚠️ {假设内容} — 推导依据: {为什么这么推导}

## 待深入

{后续阶段需要进一步挖掘的问题}

- ❓ {问题}

## 开放问题

{发现的矛盾、模糊点、需要用户额外输入的内容}

- 🔴 {问题}

---

## 变更记录

| 版本 | 日期 | 阶段 | 变更摘要 |
|------|------|------|----------|
| v0.0 | {日期} | 输入获取 | 初始创建，产品概要提取 |
```
