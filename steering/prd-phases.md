# 阶段一~四：问卷驱动的需求精炼

## 通用执行框架

每个 EXECUTE 阶段遵循相同的执行模式：

1. 读取 understanding.md，基于已有信息 + 原始输入推导本阶段内容
2. 推导结果写入 understanding.md 的"假设与推导"
3. **功能完备性前置检查**：对每个已识别的核心功能，逐项对照下方"功能完备性检查清单"，将缺失项转化为问卷问题（详见下方清单说明）
4. 只为 🔶（部分覆盖）和 ❌（未覆盖）维度生成问卷问题，同时纳入第 3 步识别的缺失项
5. ✅ 维度不生成问题，直接纳入 understanding.md 的"已明确"
6. 🔶 维度生成验证类问题（"我理解 X 是 Y，对吗？"），成本低
7. ❌ 维度生成挖掘类问题（带选项），选项根据上下文动态调整
8. 推荐选项（⭐）基于用户输入的上下文，不是固定的
9. 低复杂度时，多个 ❌ 维度可合并为一题，**但功能完备性缺失项不可合并，必须逐功能确认**

如果所有维度都是 ✅，本阶段 SKIP。问卷题目数量完全取决于覆盖度分析结果，没有固定数量。

**问卷格式**：生成问卷时必须严格遵循 `prd-question-format.md`（always 已加载）中的格式规范（Question 编号、选项格式、[Answer]: 标签、末尾提示块）。

**理解文档更新**：用户回答问卷后，更新 understanding.md 前，**Agent 必须读取 steering: `prd-understanding-template.md`** 获取更新规范和来源标注格式。

### 功能完备性检查清单（问卷生成前置检查）

**目的**：确保每个核心功能的需求细节达到"开发可直接编码"的程度，不遗漏关键业务参数。

**执行时机**：阶段二和阶段三生成问卷前必须执行。阶段二侧重场景级别（功能做什么），阶段三侧重方案级别（功能怎么做）。

对每个已识别的核心功能，逐项检查以下 7 个维度。**只检查适用的维度**（不是每个功能都需要全部 7 项）：

| # | 维度 | 检查要点 |
|---|------|----------|
| 1 | 触发方式 | 谁/什么触发此功能？（用户操作 / 系统自动 / 外部回调 / 定时 / 数据变更） |
| 2 | 触发条件 | 什么前置条件下才触发？（满足某状态、达到某阈值、特定时间窗口、用户权限等） |
| 3 | 执行动作 | 具体做什么？（不能一句话概括，必须拆解到开发可理解的粒度） |
| 4 | 数据范围 | 操作什么数据？来源和范围？（全量/增量、哪个数据源、过滤条件、数据格式） |
| 5 | 预期结果 | 成功后的输出和状态变化？（返回什么、状态变为什么、通知谁、UI 如何响应） |
| 6 | 异常处理 | 失败/超时/异常怎么办？（重试策略、降级方案、告警通知、用户提示文案） |
| 7 | 约束条件 | 有什么限制？（频率上限、数量限制、时间窗口、并发控制、权限要求、配置项及默认值） |

**使用原则**：
- Agent 先基于已有信息（understanding.md + 原始输入）推导，能推导出的写入"假设与推导"
- 推导不出的、或推导了但不确定的，才转化为问卷问题
- 问卷问题应尽量使用"推荐值 + 确认"的形式（如"刷新周期建议 5 分钟，是否合适？"），降低用户填写成本
- **推理完成后，Agent 应读取 steering: `prd-param-rules.md`（参数规则参考库），识别该功能的触发类型和功能类型，对照对应类型的常见必要参数表交叉验证，检查是否有常见参数被遗漏。** 规则库是辅助参考，Agent 仍需结合具体场景判断适用性

### 参数补充问卷机制

**执行时机**：在每个阶段的"读取答案 → 矛盾检测"之后，增加"参数完备性检测"步骤。

**流程**：
1. 读取用户的问卷答案
2. 对照功能完备性检查清单，检查用户的回答是否产生了新的未定义参数。**同时参照 steering: `prd-param-rules.md`（参数规则参考库）中对应触发类型和功能类型的常见必要参数表，辅助识别遗漏的参数**
   - 例如：用户选择了"定时刷新"，对照规则库"定时触发"类型，检查执行周期、并发控制、失败重试、数据范围等常见参数是否已明确
3. 如有缺失参数，生成参数补充问卷：
   - Workspace 模式：`.process/questionnaires/0N-xxx-参数补充.md`
   - 独立模式：`questionnaires/0N-xxx-参数补充.md`
4. 参数补充问卷每阶段最多 1 轮（与矛盾澄清的 2 轮上限独立计数）
5. 补充问卷中的问题使用"填空 + 推荐值"的形式，而非纯选项，格式遵循 `prd-question-format.md`（always 已加载）
6. 如果用户在补充问卷中仍未提供某些参数，将其写入 understanding.md 的"开放问题"，标注 `🔴 关键参数未定义: {功能名}-{参数名}`，在最终 PRD 中标注 `[待确认]`

**完整的阶段执行顺序**：生成问卷 → 等待用户填写 → 读取答案 → 矛盾检测（→ 澄清问卷，最多 2 轮）→ 参数完备性检测（→ 参数补充问卷，最多 1 轮）→ 更新 understanding.md

## 阶段一：愿景对齐

**目标**：理解"为什么做"和"做什么"，建立产品共识基线。

**关注维度**：产品定位与类型、核心问题、目标用户、成功指标、产品边界

**挖掘方向**：
- 产品定位与类型
- 核心要解决的问题
- 目标用户画像
- 成功衡量标准
- 产品边界（做什么、不做什么）
- （高复杂度额外）竞品参考、外部约束、标准规范、参考文档

---

## 阶段二：场景梳理

**目标**：把愿景拆解为具体的用户场景和故事。

**关注维度**：核心场景、场景优先级、用户旅程、异常处理

**挖掘方向**：
- 核心使用场景识别
- 场景优先级排序（Must Have / Should Have / Nice to Have）
- 每个核心场景的用户旅程（入口 → 步骤 → 结束）
- 异常情况处理
- 终端支持范围
- （高复杂度额外）角色交互关系、数据量预期、场景间关联

**特殊规则**：
- 场景列表根据阶段一的产品类型和核心问题自动推荐
- 给出推荐的流程步骤，让用户确认或修改，而不是让用户从零描述
- 每个 Must Have 场景单独深入，Should Have 简要带过
- 异常处理给出常见选项（网络、权限、空数据、校验），推荐基本必选集
- **对每个 Must Have 场景，Agent 必须按功能完备性检查清单（见通用执行框架）进行场景级检查，识别该场景涉及的关键行为和业务参数，将缺失项转化为问卷问题**

---

## 阶段三：方案细化

**目标**：把场景转化为可落地的方案要素。

**关注维度**：页面/交互方案、数据流转/状态机、权限模型

**SKIP 条件与其他阶段一致**：所有维度均为 ✅ 时才可 SKIP。不论复杂度高低，方案细节必须通过问卷确认，不能依赖 Agent 推导。

**挖掘方向**：
- 页面布局方式
- 列表页能力（搜索、筛选、排序、分页、批量操作）
- 表单交互方式
- 数据状态流转
- 权限模型
- 通知方式
- **功能关键参数定义**：对每个核心功能，按功能完备性检查清单（见通用执行框架）逐项确认，确保触发方式、触发条件、执行动作、数据范围、预期结果、异常处理、约束条件均已明确
- （高复杂度额外）数据集成方式、缓存策略

**特殊规则**：
- 状态流转根据场景自动推荐状态机
- 权限模型给出推荐（大多数 RBAC + 数据权限就够了）
- 布局方式根据产品类型推荐（管理后台→左右分栏，C端→卡片网格等）
- 如果前序阶段已经明确了某些技术选择，不要重复问
- **对每个核心功能，Agent 必须主动推导出该功能需要定义的参数清单，以"推荐值 + 确认"的形式写入问卷，而不是等用户自己想到**

---

## 阶段四：验收对齐

**目标**：定义"完成"标准和非功能要求。

**关注维度**：验收标准、非功能要求、上线策略

**挖掘方向**：
- 验收标准格式偏好（GWT / Checklist）
- 性能要求（响应时间、并发）
- 安全要求（HTTPS、防注入、加密、审计）
- 上线策略（全量/灰度/白名单/Feature Flag）
- 数据埋点需求
- （高复杂度额外）回滚方案、已知风险和依赖

**特殊规则**：
- 常规安全要求（HTTPS、防注入）和基本性能要求可以直接作为默认值写入 understanding.md，不需要问
- 如果前序阶段已经提到了某些非功能要求，不要重复问
- 低复杂度时，多个 ❌ 维度可合并为一题（如"非功能要求打包确认"）
