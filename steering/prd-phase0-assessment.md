# 复杂度评估与自适应规划（内化步骤）

这不是一个独立阶段，而是 Agent 在收到用户输入后立即执行的内部分析。其核心产出是一份执行计划，决定哪些阶段执行、哪些跳过、每个阶段的问卷深度如何。

## 核心原则：Adaptive Depth

> "Create exactly the detail needed for the problem at hand — no more, no less."

- 不要人为膨胀简单问题的细节
- 不要对复杂问题偷工减料
- 让输入的完善程度和问题特征自然驱动深度

## 执行流程

### 前置条件：understanding.md 已创建

Phase 0 执行前，understanding.md 应已在输入获取阶段创建（**Agent 必须读取 steering: `prd-understanding-template.md`** 获取模板）。初始内容包括：
- 版本号 `v0.0`
- 产品概要（从用户输入中提取一段话概述）
- "已明确"区域为空（Phase 0 会填入 ✅ 维度内容）
- "假设与推导"区域为空
- "待深入"区域为空
- 变更记录第一条：`v0.0 | {日期} | 输入获取 | 初始创建，产品概要提取`

Phase 0 完成后，understanding.md 版本更新为 `v0.1`，✅ 维度内容写入"已明确"，Agent 推导的内容写入"假设与推导"。

### Step 0：读取产品全景（Workspace 模式）

Workspace 模式下，Agent 在开始覆盖度分析前，**必须**读取 `workspace/knowledge/product/landscape.md`（如存在）：
- 了解系统现状、已有模块、技术栈，作为推导的背景上下文
- 如果新需求涉及已有模块，记录模块名，后续阶段按需读取 `modules/{模块名}.md` 深入了解
- 如果 landscape.md 不存在（全新 workspace），跳过此步骤

### Step 1：输入覆盖度分析

Agent 逐维度扫描用户输入，判断每个维度的覆盖状态：

| 维度 | 对应阶段 | 覆盖判定标准 |
|------|----------|-------------|
| 产品定位与类型 | 阶段一 | 明确说了做什么类型的产品 |
| 核心问题 | 阶段一 | 明确说了要解决什么问题 |
| 目标用户 | 阶段一 | 明确说了给谁用 |
| 成功指标 | 阶段一 | 有可衡量的目标或 KPI |
| 产品边界 | 阶段一 | 明确说了做什么、不做什么 |
| 核心场景 | 阶段二 | 列出了主要使用场景，**且每个场景的关键行为有明确的规则描述**（触发方式、执行动作、预期结果至少三项明确）。仅列出场景名称但缺少行为细节的，判定为 🔶 |
| 场景优先级 | 阶段二 | 场景有优先级排序 |
| 用户旅程 | 阶段二 | 描述了核心流程步骤，**且每个步骤的输入输出和业务规则有明确定义**。仅列出步骤名称但缺少输入输出的，判定为 🔶 |
| 异常处理 | 阶段二 | 列出了异常场景**且每个异常有明确的处理策略**（不能只说"需要处理异常"）。仅提及"需考虑异常"但无具体策略的，判定为 🔶 |
| 页面/交互方案 | 阶段三 | 有页面布局和交互描述，**且涉及的功能行为（如刷新、提交、筛选）有明确的触发条件和业务规则**。仅有线框图但缺少行为规则的，判定为 🔶 |
| 数据流转/状态机 | 阶段三 | 描述了**完整的**状态流转（每个状态的进入条件、流出条件、触发动作都明确）。仅列出状态名称但缺少流转条件的，判定为 🔶 |
| 权限模型 | 阶段三 | 描述了角色和权限 |
| 验收标准 | 阶段四 | 有明确的验收条件 |
| 非功能要求 | 阶段四 | 提到了性能、安全等要求 |
| 上线策略 | 阶段四 | 描述了上线方式 |

每个维度标记为：
- ✅ 已覆盖 — 输入中有明确描述，无需提问，直接写入 understanding.md 的"已明确"（标注 `[来源: 输入直接纳入]`，格式见 `prd-understanding-template.md`，**Agent 必须读取 steering: `prd-understanding-template.md`** 获取来源标注规范）
- 🔶 部分覆盖 — 有提及但不够具体，生成验证类问题（"我理解 X 是 Y，对吗？还有补充吗？"）
- ❌ 未覆盖 — 完全没提到，需要挖掘类问题

### Step 2：阶段选择（EXECUTE / SKIP）

基于覆盖度分析，决定每个阶段是否执行：

**SKIP 条件**：该阶段所有维度均为 ✅ 已覆盖。
- 跳过时，Agent 将该阶段的所有已覆盖内容直接写入 understanding.md 的"已明确"
- 在聊天中告知用户："阶段 X 的内容你已经描述得很清楚了，我直接纳入理解，不再提问。"

**EXECUTE 条件**：该阶段有任何 🔶 或 ❌ 维度。
- 执行时，问卷只包含 🔶 和 ❌ 维度的问题，✅ 维度不生成问题

**特殊规则**：
- 如果所有四个阶段都 SKIP，Agent 直接进入撰写就绪检查（见 `prd-workflow-core.md`），但在此之前必须展示完整理解让用户确认：
  ```
  📋 你的输入已覆盖所有 15 个维度，四个阶段全部跳过，无需额外提问。
  
  我已将所有内容提取到 understanding.md（v0.1），请查看确认理解是否准确。
  特别关注"已明确"中标注 [来源: 输入直接纳入] 的条目，确保我没有曲解原意。
  
  确认后我开始撰写最终 PRD。如果有需要调整的地方，现在告诉我。
  ```

### Step 3：复杂度判定

综合以下因素判定复杂度（低/中/高），影响 EXECUTE 阶段的问卷深度：

| 因素 | 考量 |
|------|------|
| 请求清晰度 | 输入的完善程度（覆盖度分析结果） |
| 问题复杂度 | 单一功能 vs 多模块协作 vs 跨系统 |
| 影响范围 | 单组件 vs 多组件 vs 系统级 |
| 风险等级 | 出错影响大小 |
| 已有上下文 | 有详细参考 vs 全新领域 |
| 用户偏好 | 用户是否表达了对简洁或详细的偏好 |

### Step 4：生成执行计划

在聊天中告知用户：

```
我分析了你的需求，以下是执行计划：

📊 复杂度：{低/中/高}
📋 输入覆盖度：{N}/{总数} 个维度已覆盖

| 阶段 | 决策 | 原因 |
|------|------|------|
| 阶段一：愿景对齐 | {EXECUTE/SKIP} | {简要原因，如"产品定位和用户已明确，但成功指标未提及"} |
| 阶段二：场景梳理 | {EXECUTE/SKIP} | {简要原因} |
| 阶段三：方案细化 | {EXECUTE/SKIP} | {简要原因} |
| 阶段四：验收对齐 | {EXECUTE/SKIP} | {简要原因} |

{如果有 SKIP 的阶段}
已跳过阶段的内容我已纳入理解文档，你可以随时查看和修改。

如果你对执行计划有异议（比如某个阶段想改为 EXECUTE 或 SKIP），现在可以告诉我调整。
确认没问题的话，我们开始第一个阶段。
```

**用户确认窗口**：展示执行计划后，必须等待用户确认或调整。用户可能：
- 直接确认（"ok"/"没问题"/"开始"）→ 进入第一个 EXECUTE 阶段
- 要求调整某阶段的 EXECUTE/SKIP 决策 → 修改执行计划后重新展示
- 要求调整复杂度判定 → 重新评估后更新执行计划

用户确认后，立即生成第一个 EXECUTE 阶段的问卷并告知用户。

**下一步衔接**：用户确认执行计划后，**Agent 必须读取 steering: `prd-phases.md`** 获取对应阶段的维度和挖掘方向，按 `prd-question-format.md`（always 已加载）的格式生成第一轮问卷。

当阶段被标记为 EXECUTE 时，问卷深度由两个因素共同决定：

1. **维度覆盖度**：✅ 维度不生成问题，🔶 生成验证类问题（成本低），❌ 生成挖掘类问题
2. **复杂度等级**：影响 ❌ 维度的挖掘深度

| 复杂度 | ❌ 维度处理 | 🔶 维度处理 |
|--------|-----------|-----------|
| 低 | 精简挖掘（1题覆盖多维度） | 打包验证（多个🔶合并为1题） |
| 中 | 标准挖掘（每维度1题） | 逐个验证 |
| 高 | 深入挖掘（关键维度可拆多题） | 逐个验证 + 追问细节 |

**结果**：同一个阶段，输入完善的文档可能只需 1-2 题验证，而模糊的 idea 可能需要 7-9 题挖掘。

## 示例

### 示例 A：用户提供了详尽的 PRD 初稿

输入覆盖度：13/15 维度已覆盖，缺"上线策略"和"数据埋点"

```
执行计划：
- 阶段一：SKIP（愿景已完整描述）
- 阶段二：SKIP（场景和流程已详细列出）
- 阶段三：SKIP（交互方案和权限已明确）
- 阶段四：EXECUTE（2题 — 上线策略和埋点需求未提及）
```

### 示例 B：用户提供了一句话 idea

输入覆盖度：1/15 维度已覆盖（只有产品类型）

```
执行计划：
- 阶段一：EXECUTE（4题 — 产品类型已知，需挖掘问题、用户、指标、边界）
- 阶段二：EXECUTE（6题 — 全部需挖掘）
- 阶段三：EXECUTE（5题 — 全部需挖掘）
- 阶段四：EXECUTE（4题 — 全部需挖掘）
```

### 示例 C：用户提供了中等完善度的文档

输入覆盖度：8/15 维度已覆盖

```
执行计划：
- 阶段一：EXECUTE（2题 — 成功指标和边界需确认）
- 阶段二：SKIP（场景和流程已完整）
- 阶段三：EXECUTE（3题 — 权限和状态流转需明确）
- 阶段四：EXECUTE（3题 — 验收标准和上线策略需明确）
```
