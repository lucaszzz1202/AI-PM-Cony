# PRD 工作流变更管理

当用户在工作流执行过程中请求变更时，按以下规范处理。

**注意：以下规范中的文件路径使用通用名称。Agent 根据当前工作模式（Workspace / 独立）自动映射到实际路径。如需查看映射关系，Agent 必须读取 steering: `prd-file-specs.md`。**

**注意：以下规范涉及多处 understanding.md 更新操作。任何更新 understanding.md 的操作，Agent 必须遵循 `prd-understanding-template.md` 的更新规范和来源标注格式（fileMatch 自动加载；如未自动加载，Agent 必须读取 steering: `prd-understanding-template.md`）。**

## 变更类型与处理

### 1. 回退到之前的阶段

用户说"回到阶段一"、"重新梳理场景"等。

处理：
1. 确认请求："你想回到阶段{N}重新调整，这会影响后续阶段的产出，确认？"
2. 备份当前 understanding.md 为 `understanding.backup.md`
3. **标注待重新验证**（不真正删除已有内容）：在 understanding.md 的"已明确"区域中，将阶段{N}及后续阶段产出的条目标注为 `[待重新验证: 回退到阶段{N}]`。具体规则：
   - 标注 `[来源: 阶段{N}问卷]` 及后续阶段来源的条目 → 追加 `[待重新验证: 回退到阶段{N}]`
   - 标注 `[来源: 输入直接纳入]` 的条目 → 保持不变（这些来自用户原始输入，不受回退影响）
   - 标注 `[来源: Agent推导]` 且推导依据涉及被回退阶段的条目 → 追加 `[待重新验证: 回退到阶段{N}]`
4. 重新生成该阶段问卷（**Agent 必须读取 steering: `prd-phases.md`** 获取维度和挖掘方向，按 `prd-question-format.md`（always 已加载）格式生成）
5. 后续阶段的问卷和产出标记为待重做
6. 在 understanding.md 变更记录中记录回退原因（**Agent 必须读取 steering: `prd-understanding-template.md`** 获取更新规范和来源标注格式）
7. 用户回答新问卷后，将确认的条目去掉 `[待重新验证]` 标注，修正错误的条目则更新内容并标注新来源

### 2. 跳过当前阶段

有两种跳过方式，处理逻辑不同：

#### 2a. 系统自动 SKIP（Adaptive Depth）

在复杂度评估阶段，Agent 基于覆盖度分析自动决定 SKIP。此时：
- 该阶段所有 ✅ 维度的内容已写入 understanding.md 的"已明确"，标注 `[来源: 输入直接纳入]`
- 状态追踪标记为 `⏭️ 跳过（输入已覆盖）`
- 不需要用户确认（执行计划中已展示，用户可以在那时提出异议）
- 质量自检时会做 SKIP 阶段内容复核

#### 2b. 用户主动跳过

用户在阶段执行过程中说"跳过这个阶段"、"不需要方案细化"等。

处理：
1. 说明影响："跳过阶段{N}意味着{缺失内容}，最终 PRD 的{章节}会比较简略。确认？"
2. 用户确认后，在 understanding.md 中记录跳过决策和原因
3. 状态追踪标记为 `⏭️ 跳过（用户主动跳过）`
4. Agent 尽可能基于已有信息推导该阶段的内容，写入 understanding.md 的"假设与推导"
5. 进入下一阶段

### 3. 修改已确认的决策

用户说"目标用户改一下"、"权限模型要换成 ABAC"等。

处理：
1. 评估影响范围：这个变更会影响哪些后续决策？
2. 如果影响小（仅改一个点，不影响其他决策）：直接更新 understanding.md，继续当前阶段
3. 如果影响大（牵一发动全身）：建议回退到相关阶段重新走
4. 在 understanding.md 变更记录中记录变更内容和原因

### 4. 增加新需求

用户说"再加个功能"、"还需要支持 XX"等。

处理：
1. 判断新需求属于哪个阶段的范畴
2. 如果当前阶段还没结束：直接纳入当前阶段处理
3. 如果当前阶段已过：
   - 小需求：直接更新 understanding.md，在最终撰写时纳入
   - 大需求：建议回退到对应阶段重新走
4. 更新 understanding.md

### 5. 对 PRD 终稿的修改

用户拿到最终 PRD 后说"这里要改"。

处理：
1. 小改（措辞、格式、补充细节）：直接改最终 PRD
2. 中改（调整某个功能的设计）：改最终 PRD + 更新理解文档 + 重跑质量自检。**Workspace 模式下同步更新 requirements/features.md**
3. 大改（新增/删除功能、改变产品方向）：建议回退到对应阶段重新走
4. 任何修改后都要重跑质量自检

### 6. 暂停与恢复

用户说"先到这里"、"明天继续"等。

**暂停处理：**
1. 确保理解文档（understanding.md）是最新状态
2. 更新状态追踪文件（Workspace 模式: state.md / 独立模式: 评审概览.md）的中断点信息（精确记录当前位置和下一步）
3. 确保状态追踪文件的阶段进度是最新状态
4. **Workspace 模式额外步骤**：追加 session.log.md 暂停记录
5. 在聊天中告知当前进度："目前完成到阶段{N}，下次继续时从阶段{N+1}开始。所有进度已保存。"

**恢复处理：**

恢复由 `prd-workflow-core.md` 的"会话连续性"机制自动触发。每次新对话开始时，Agent 会自动扫描 `workspace/` 和 `.prd-refine/` 目录，检测到进行中的项目后执行完整的恢复流程（读取状态追踪 → 读取 understanding → 扫描文件 → 加载前序产物 → 展示恢复摘要 → 用户确认后继续）。

用户无需手动说"继续"，Agent 会主动检测并提示。

## 通用原则

1. **先确认再动手** — 任何破坏性变更都要用户明确确认
2. **说明影响** — 让用户理解变更的代价再做决定
3. **备份再改** — 回退或大改前备份 understanding.md
4. **记录变更** — 所有变更都记入 understanding.md 的变更记录
5. **质量自检** — 涉及最终 PRD 的修改，改完重跑质量自检
