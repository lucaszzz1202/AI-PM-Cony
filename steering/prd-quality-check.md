# PRD 质量自检

## 目标

在交付前对最终 PRD 做一轮系统性质量检查，发现遗漏和不一致，自动补全或标注待确认。

## 执行时机

原型设计环节之后、输出回填之前。

**检查对象**：
- Workspace 模式: `requirements/final.md`
- 独立模式: `output/完整PRD.md`

**报告输出**：
- Workspace 模式: 生成 `.process/质量报告.md`
- 独立模式: 生成 `output/质量报告.md`

## 检查维度

### 0. SKIP 阶段内容复核（Adaptive Depth 专项）

当有阶段被 SKIP（输入已覆盖）时，这些阶段的内容是从用户输入中直接提取的，没有经过问卷验证。质量自检需要额外复核：

| 检查项 | 规则 |
|--------|------|
| 直接纳入的内容是否准确提取 | 对比原始需求文件和 understanding.md 中标注 `[来源: 输入直接纳入]` 的条目，确认没有曲解原意 |
| 直接纳入的内容是否充分 | 检查原始输入中是否有相关信息被遗漏未纳入 |
| 跨阶段一致性 | SKIP 阶段直接纳入的内容与 EXECUTE 阶段问卷确认的内容之间是否有矛盾 |
| 推导合理性 | Agent 基于直接纳入内容做的推导（标注 `[来源: Agent推导]`）是否合理 |

如果发现问题，标记为 ⚠️ 建议补充，在报告中说明："此内容来自 SKIP 阶段的直接纳入，建议用户确认。"

### 1. 完整性检查

| 检查项 | 规则 |
|--------|------|
| 每个功能需求（FR）是否都有对应的用户故事（US） | FR 和 US 必须一一对应 |
| 每个用户故事是否都有验收标准（AC） | 没有 AC 的故事无法验收 |
| 验收标准是否可测试 | AC 必须包含具体的可验证条件，不能是"系统正常工作" |
| 所有产品角色是否都在用户故事中出现 | §5 定义的角色必须在 §7 中有对应场景 |
| 非功能需求是否有量化指标 | "性能要好"不行，"响应时间 < 500ms"才行 |
| 状态流转是否完整 | 每个状态都有进入条件和流出路径，无死胡同 |
| 异常场景是否覆盖 | 每个核心流程至少覆盖：网络异常、权限不足、数据为空、输入校验 |

### 1.5 功能完备性检查（开发可执行性）

**核心判断标准：如果一个开发拿到这个 FR，能否不问任何人就开始编码？不能就是 🔴。**

对每个 FR，逐项检查以下维度（仅检查适用项，参照 `prd-phases.md` 中的功能完备性检查清单）：

| 检查项 | 规则 | 严重度 |
|--------|------|--------|
| 触发方式是否明确 | 必须说明是用户操作 / 系统自动 / 外部调用 / 定时触发，不能缺失 | 🔴 |
| 触发条件是否具体 | 不能是"满足条件时触发"，必须写明具体条件（如"当库存低于安全库存值时"） | 🔴 |
| 执行动作是否可编码 | 开发读完能直接写代码，不需要再问产品"具体做什么" | 🔴 |
| 数据范围是否定义 | 操作的数据来源、范围、过滤条件必须明确 | 🔴 |
| 预期结果是否明确 | 成功输出、状态变化、通知对象必须定义 | ⚠️ |
| 异常处理是否覆盖 | 至少覆盖失败、超时两种异常的处理策略 | ⚠️ |
| 约束条件是否定义 | 涉及频率、数量、时间的功能必须有具体数值或明确标注 `[待确认]` | 🔴 |
| Business Rules 表是否完整 | 每条规则必须包含"条件-动作-参数"三要素，缺少任何一个要素标记为不完整 | 🔴 |
| 常见必要参数覆盖 | **参照 steering: `prd-param-rules.md`（参数规则参考库）**，识别该 FR 的触发类型和功能类型，对照对应类型的常见必要参数表，检查是否有常见参数缺失且未标注 `[待确认]`。规则库是辅助参考，仍以上述原则判断为最终裁定标准 | ⚠️ |

### 1.6 understanding.md 覆盖率检查

扫描 understanding.md "已明确"区域的所有条目，检查每条是否在 final PRD 中有对应体现：

| 检查项 | 规则 | 严重度 |
|--------|------|--------|
| 功能需求类条目 | 必须在 §7 对应 FR 中体现 | 🔴 |
| 业务规则类条目 | 必须在对应 FR 的 Business Rules 表中体现 | 🔴 |
| 非功能需求类条目 | 必须在 §8 中体现 | ⚠️ |
| 假设/约束类条目 | 必须在 §11 中体现 | ⚠️ |
| 背景/上下文类条目 | 应在 §2 或 §6 中体现 | ⚠️ |

> 覆盖率低于 100% 时，列出所有未覆盖条目及其分类和严重度。

### 2. 一致性检查

| 检查项 | 规则 |
|--------|------|
| §4 产品范围 vs §7 功能需求 | 功能不能超出范围，范围内的 Must Have 不能遗漏 |
| §5 角色权限 vs §7 功能描述 | 功能中涉及的权限必须在角色定义中有体现 |
| §2 成功指标 vs §7 功能 + §8 非功能 | 指标必须有对应的功能或埋点支撑 |
| 术语一致性 | 同一概念在全文中使用同一术语，不能一会儿叫"订单"一会儿叫"工单" |
| 优先级一致性 | 用户故事的优先级和功能需求的优先级必须一致 |
| FR 间联动一致性 | 多个 FR 操作同一数据实体或共享触发条件时，检查：① 业务规则是否一致（如同一字段的校验规则在不同 FR 中定义相同）② 是否有遗漏的联动场景（如 FR-A 修改了某状态，依赖该状态的 FR-B 是否感知并处理）③ 共享参数的定义是否统一（如同一个"刷新周期"在多处引用时值一致） |
| 与现有模块冲突检查（Workspace 模式） | 读取 `landscape.md`，检查新功能是否与已有模块功能重复或冲突 |

### 3. 专业性检查

| 检查项 | 规则 |
|--------|------|
| 需求是否可实现 | 不包含技术上不可行或极端模糊的要求 |
| 是否有隐含假设未说明 | 所有假设必须在 §11 中显式列出 |
| 依赖是否完整 | 涉及的外部系统、第三方服务必须在 §10 中列出 |
| 业务规则是否明确 | 不能有"根据情况处理"这类模糊表述 |
| 边界条件是否定义 | 数值型字段的最大最小值、文本的长度限制、列表的上限 |

## 执行方式

### 自动检查

Agent 逐项检查最终 PRD，生成检查报告：

**独立模式**生成 `output/质量报告.md`：

```markdown
# PRD 质量检查报告

> 检查时间: {日期}
> 检查版本: 最终 PRD v1.0

## 检查结果摘要

- ✅ 通过: {N} 项
- ⚠️ 建议补充: {N} 项
- 🔴 必须修复: {N} 项

## 详细结果

### ✅ 通过项
- {检查项}: {通过原因}

### ⚠️ 建议补充
- {检查项}: {问题描述} → 建议: {具体建议}

### 🔴 必须修复
- {检查项}: {问题描述} → 需要: {需要用户提供什么信息}
```

### 自动补全

对于"建议补充"类问题，根据类型区分处理：

**可直接自动补全（不需要用户确认）：**
- 缺少的异常场景 → 基于行业经验补充常见异常处理（网络异常、权限不足、空数据、输入校验）
- 缺少的术语定义 → 从上下文提取并补充到 §1.5
- 格式规范化 → 统一术语、修正格式不一致
- 缺少的依赖声明 → 从功能描述中提取明显的外部依赖补充到 §10
- 缺少的假设声明 → 从 understanding.md 的已确认假设迁移到 §11（**Agent 必须读取 steering: `prd-understanding-template.md`** 获取来源标注规范）

**需标注"Agent 补充，请确认"后自动补全：**
- 缺少量化指标的非功能需求 → 补充行业通用基线值（如响应时间 < 500ms），但标注 `[Agent 建议值，请确认]`
- 缺少的边界条件 → 补充常见默认值（如列表上限 1000 条），但标注 `[Agent 建议值，请确认]`

**不可自动补全（必须用户提供）：**
- 具体的业务规则数值（如审批金额阈值、折扣比例）
- 安全合规的具体要求（如数据加密等级、合规标准编号）
- 上线策略的具体参数（如灰度比例、白名单范围）
- 涉及商业决策的内容（如定价、SLA 承诺）
- **功能缺少的关键业务参数**（如触发条件、执行频率、数据范围等——这些是开发必须知道的信息，不能由 Agent 猜测，必须标 🔴 由用户提供）

### 用户确认

对于"必须修复"类问题和标注了"Agent 建议值，请确认"的补全项，在聊天中告知用户，等待确认或修改后更新。

## 质量门禁

🔴 项为 0 时才允许进入输出回填环节。如果有 🔴 项，必须解决后重新检查。

**Workspace 模式额外检查**：
- 检查 `requirements/features.md` 是否与 `requirements/final.md` §7 一致。如果质量自检修改了 §7 功能需求（新增/删除/修改 FR）或 §7.3 用户故事/验收标准，必须同步更新 features.md。其他章节的修改（如 §1.5 术语、§8 非功能需求、§11 假设）不需要同步 features.md。
- 检查 features.md 中每个 FR 的"涉及模块"是否已标注。如果存在 `workspace/knowledge/product/landscape.md`，验证标注的模块名称与全景中的功能模块清单一致（模块名拼写匹配、不引用不存在的模块）。标注为 `🆕 新增模块` 的条目不做匹配校验，但需确认模块名称合理且不与已有模块重名。

**下一步衔接**：🔴 项清零后，进入输出回填环节。如果用户选择回填 Confluence，**Agent 必须读取 steering: `prd-confluence.md`** 获取回填流程。Workspace 模式下，回填后 **Agent 必须读取 steering: `prd-workspace-knowledge.md`** 依次执行知识沉淀和阶段流转。
