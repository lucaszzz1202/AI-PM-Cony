# 问卷交互规范

## 核心规则

- **禁止在聊天中直接提问** — 所有问题必须写入独立的问卷文件
- **每个问题必须包含 Other 选项** — 作为最后一个选项
- **选项必须有意义** — 不凑数，2-5 个有意义选项 + Other
- **等待用户确认完成** — 用户说"完成"/"done"后才读取答案
- **矛盾必须解决** — 检测到矛盾时生成澄清问卷，解决后才继续

## 问题格式

```markdown
## Question [编号]

[清晰、具体的问题描述]

A) [选项A — 推荐时标注 ⭐]
B) [选项B]
[...2-5个有意义选项...]
X) Other (please describe after [Answer]: tag below)

💡 推荐理由: [一句话]

[Answer]:
```

## 问卷文件末尾固定提示

每份问卷文件的末尾必须附加以下提示块，确保用户知道如何通知 Agent：

```markdown
---

✏️ 填写完成后，请在聊天中回复"完成"或"done"，我会读取你的答案并继续。
如果某题不确定，可以先空着，回复"完成"后我会提示你补充。
```

## 用户回答方式

- 单选：`[Answer]: A`
- 多选：`[Answer]: A,C,D`
- Other：`[Answer]: X - 补充说明`

## 矛盾检测

读取答案后必须检查逻辑一致性。常见矛盾：
- 范围矛盾：目标用户与场景不匹配
- 指标矛盾：成功指标与产品类型不匹配
- 优先级矛盾：Must Have 依赖了"不做"的能力
- 技术矛盾：终端选择与使用场景矛盾

检测到矛盾时创建 `questionnaires/0N-xxx-澄清.md`，格式同问卷，但需说明矛盾原因。

### 澄清轮次上限

同一阶段最多 2 轮矛盾澄清（即最多生成 `0N-xxx-澄清.md` 和 `0N-xxx-澄清2.md`）。如果 2 轮后仍有未解决的矛盾：
1. 将未解决的矛盾写入 understanding.md 的"开放问题"区域，标注 `🔴 矛盾未解决: {矛盾描述}`
2. 在聊天中告知用户："有 {N} 个矛盾点经过两轮澄清仍未完全解决，已记录为开放问题。我会在最终 PRD 中标注'待确认'，你可以在后续阶段或终稿审查时再处理。"
3. 继续推进到下一步骤，不再阻塞流程

## 错误处理

| 情况 | 处理 |
|------|------|
| `[Answer]:` 为空 | 提示补充 |
| 无效字母 | 提示使用有效选项 |
| 模糊回答 | 提示选字母或选 Other |
